# About these design docs

*Last revised: 2026-06-04.*

This project is at the pre-code stage. Before any application code exists, four prose docs (`1.`–`4.`) define what the product does, how its identity model works, what its data looks like, and which services run it. This doc explains what those four docs are, why they exist in this form, and what industry practice they correspond to. Read it first if you're new to the repo; skim it as a refresher if you're returning.

## The four docs

| # | File | What it pins down |
|---|---|---|
| 1 | `RFC-1 product-questions.md` | The three primitives — discovery, temporal, differential — and the scopes (corpus / document / clause) they accept. Defines the customer-visible contract before any endpoint exists. |
| 2 | `RFC-2 clause-alignment.md` | How a clause keeps its identity across versions of a document under renumbering. The technical invariant the entire change-detection pipeline rests on. |
| 3 | `RFC-3 database-design.md` | Performance target, scale assumptions, append-only / clause-granular principles, and database choice direction. Schema details deferred. |
| 4 | `RFC-4 services.md` | The three deployable services, what each does and explicitly does not do, and the cross-cutting principles — including the two-axis multi-tenant isolation guarantee. |

Each doc has an **Open questions** footer where decisions still pending are listed honestly rather than hidden.

## What this practice is called

Two recognisable industry traditions describe these docs. They are not mutually exclusive — what we have here is a hybrid.

### Design docs / RFCs (the primary frame)

*RFC* — *Request for Comments* — is the term of art, borrowed from the IETF, where it has named the documents defining internet standards since 1969. In engineering practice it now describes an internal narrative document that captures a system's design and rationale, written before or alongside code and circulated to peers for comment before being accepted. Google publishes a design-doc template that is closely emulated across the industry; Stripe, Cloudflare, Square, and most engineering orgs at scale have analogous practices. Amazon's "Working Backwards" PR-FAQ and its 6-page narrative-memo culture are the same family.

The defining characteristic is prose-shaped reasoning about *why*, not specification of *what to type*. The intended reader is an engineer who will build, maintain, or extend the system — including a future version of the author who has forgotten the context.

Our four docs are a **linked RFC chain at the architectural level**. Each builds on the previous one; each occupies a chapter-sized scope (e.g. "the identity model", "the services") rather than a single-decision scope. This is coarser than the typical per-feature RFC, but it matches the grain of decisions that are expensive to revisit later.

### Architecture Decision Records (ADRs) (the secondary frame)

Smaller, atomic decision records popularised by Michael Nygard around 2011. A canonical ADR captures *one* decision with three fields: context, decision, consequences. ADRs live in-repo, are numbered, and are immutable once accepted — a superseded ADR is replaced by a new ADR, not edited in place.

Our docs are coarser than canonical ADRs (chapter-grained, not decision-grained), but they share two things:

- The discipline of recording *why* alongside *what*.
- The **Open questions** footer style, which plays the role of an ADR's `Status:` field — honest about what's still in flight.

Where ADRs would be additive to what we have today is for sub-decisions that emerge during implementation (e.g. "we picked RLS-only over RLS + repository pattern because …"). That's a future option; the current docs are the architectural backbone, and ADRs would slot in as more granular siblings if and when we need them.

## Why this approach, on this project

Specific principles from this project's setup that make a pre-code design-doc chain worthwhile:

- **Bus factor zero.** Decisions and rationale need to survive personnel change. Code captures *what*; docs capture *why*. (See the project's `design priorities` memory.)
- **Foundational invariants are expensive to retrofit.** Clause-level vs document-level change detection, stable clause identity, multi-tenant isolation, subscription gating — getting any of these wrong is a much larger cost than getting an endpoint shape wrong. Spending time before code on the things that are expensive to undo is good leverage.
- **Demo as accountability.** The product is heading toward a public demo (2026-06-08). Public exposure forces the design to actually work end-to-end, which forces the docs to be honest end-to-end.

## How docs evolve

- **`*Last revised:*` header** at the top of every doc in this chain. Update it whenever the body changes meaningfully (not for typo fixes).
- When a decision in an existing doc is superseded, **prefer editing in place** and noting the change in the relevant section, rather than starting a new doc — unless the change is large enough to warrant a new chapter.
- When a doc *is* superseded wholesale, archive per the global rule: `git mv` it into `docs/archive/`, add a `**Status:** superseded by [...](...) (YYYY-MM-DD).` header at the top, update inbound links.
- **Open questions are first-class.** Leave them open until they're actually resolved. Resolving one is itself a doc-update event.

## Honest caveats and how this project addresses them

Three well-documented failure modes of design-doc-heavy practice, named so we can defend against them:

### Big-Design-Up-Front (BDUF)

The agile critique: over-specifying abstractions before contact with code locks in wrong shapes. The historical reputation of BDUF was forged in an era of slow design-to-code cycles — months of design, months more before feedback. In an AI-assisted workflow where revising a doc and regenerating the affected code is a same-day operation, the cost asymmetry that justified the BDUF critique no longer holds in the same form.

The grain of these docs is also chosen deliberately to mitigate BDUF: they specify *invariants and boundaries* (identity model, multi-tenant axes, service responsibilities), not class hierarchies or function signatures. That's the level of decision worth making in prose.

### Doc rot

Docs that aren't kept current become traps — worse than no docs, because they confidently mislead. Defences in use here:

- The `*Last revised:*` header makes rot visible.
- Doc updates are expected to be part of any meaningful code change, not a separate chore.
- Open questions get resolved (and the resolution gets written down) rather than quietly forgotten.

### False confidence

Prose that *sounds* coherent is not the same as a design that *works*. Two complementary defences:

- **Senior-engineer review.** Catches judgment-shaped errors: this is over-engineered, this maps poorly to the domain, this won't survive a real customer, this assumes resources we don't have. Humans are good at the "does this actually fit reality" question.
- **AI review.** Catches consistency-shaped errors: contradictions between docs, unbacked claims, missed edge cases, drift between a principle stated in one doc and a mechanism described in another. AI is reliable at this *when given enough context* — review the full 1→4 chain together, not single docs in isolation. Lacking context, AI will confidently agree with bad designs.

The two reviews are complementary, not substitutes. Use both.
