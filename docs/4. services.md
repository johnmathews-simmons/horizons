# Services: what they do, what they don't, and how they fit

*Last revised: 2026-06-04.*

The earlier docs in this chain define *what the product answers* (`1. product-questions.md`), *how clause identity survives across versions* (`2. clause-alignment.md`), and *what the database looks like* (`3. database-design.md`). This doc names the deployable services on top of that database, what each one is responsible for, what it deliberately is not responsible for, and the high-level mechanics of how they cooperate.

Three deployable services sit over the shared Postgres + Blob storage from doc 3:

```
                    ┌─────────────────┐
   Lawstronaut ──▶  │  Ingestion      │
   (upstream API)   │  worker         │
                    └────────┬────────┘
                             │ writes
                             ▼
                    ┌─────────────────┐
                    │ Postgres + Blob │
                    └────────┬────────┘
                             │ reads
                             ▼
                    ┌─────────────────┐
                    │  Public API     │
                    │  (HTTP, REST)   │
                    └─────────────────┘
                       ▲           ▲
                       │           │
                ┌──────┴──┐   ┌────┴────────────┐
                │ Webapp  │   │   Customer      │
                │ (SPA)   │   │   API access    │
                └─────────┘   └─────────────────┘
```

All client interaction — through the webapp or programmatic — goes through the **same** public API. There is no internal back-channel.

## Cross-cutting principles

These apply to every service and every endpoint. Violating any of them is a design bug, not a tuning question.

### Multi-tenant isolation is the core architectural principle

Isolation has **two axes**, both load-bearing and both enforced from day one.

**1. Cross-client isolation of private state.**
- Per-user state — watchlists, saved queries, dashboard preferences, alerts, and subscriptions (defined below) — is **private** to that user.
- **No client can observe any state that belongs to another client.** Not the existence of other users, not their watchlists, not their query history, not aggregate stats that could leak per-client behaviour.

**2. Subscription scoping of corpus access.**
- The legal corpus (documents, versions, clauses, change events) is **global** and centrally maintained. But each client buys a **subscription** that defines which slice of the corpus they can query — expressed as a set of jurisdictions and/or sectors (and possibly other filters; see §"Open questions").
- A client subscribed to UK laws cannot see change events for EU documents, even though those rows live in the same tables. **A subscription breach is a trust breach** and is treated with the same severity as a cross-client leak.
- Subscriptions are a private-state row owned by `admin`: clients can read their own subscription; only `admin` can write or modify it. A client's watchlist must be a subset of their subscription scope; this is validated on watchlist writes.

Two roles, recorded on the user record: `client` and `admin`. `admin` is for us, the operators, and unlocks system health, subscription management, and the ability to view any client's private state *and* full corpus for support purposes. `client`-to-`client` visibility remains zero on both axes regardless of any flag. The worker is corpus-global and knows nothing about users or subscriptions.

Enforcement lives at the API layer and uses defence in depth — see §"Public API service / How".

### API responsiveness is non-negotiable

The public API must stay responsive even while ingestion is in flight. This is a commercial-grade product, and customers polling for discovery cannot be made to wait for an ingestion process to finish.

Mechanism:
- Ingestion runs in a **separate container** with its own resource envelope; an ingestion burst cannot starve the API of CPU or connections.
- API and worker share Postgres but operate on **disjoint hot paths**: the worker writes `documents`, `document_versions`, `clauses`, `change_events`; the API reads them. Per-user tables are read/written only by the API.
- A read replica is a tunable lever (lift read traffic off the writer) — not a redesign. Out of scope until measured.

### One public API surface

The webapp is a customer of the same REST API every external customer uses. No "internal" endpoints, no admin-only host, no service-to-service bypass for the SPA. This keeps the contract honest and forces multi-tenant boundaries to be exercised by our own UI every day.

### Configuration over code

The curated document set, jurisdictions, sectors, polling cadence per document, and tuning thresholds (per doc 2) are **data**, not code. Adding a document to the curated set or changing a poll interval must not require a redeploy. This matches the "Configuration over code" principle already stated in doc 3 and in `CLAUDE.md`.

## Ingestion service

A scheduled worker that keeps the database current with Lawstronaut.

### Does

- Polls Lawstronaut for every document in the **operator-curated set** on its configured cadence.
- For each poll of document *D*: fetches the current head version, normalises the content, and computes a content hash. Then:
  - **Unchanged** (hash equals the live version's stored hash): extends that live version's `valid_to` from the previous poll timestamp to the current one. No new row, no clause work, no events.
  - **Changed**: closes the previous version's `valid_to`, ingests the new version (markdown to Blob, row in `document_versions` with a fresh `valid_from = now`, `valid_to = now`), parses clauses, runs the clause-alignment pipeline against the previous version (per doc 2), and writes the resulting `change_events` rows.
- Writes per-poll telemetry: timestamp, HTTP status, upstream latency, content hash, decision (`unchanged` / `changed` / `failed`), worker run id. This is the substrate for the health-and-coverage endpoints (see "Public API service / Does") and the dashboard's "last scanned" / "last change" stats.
- Retries on transient upstream errors with backoff. On persistent failure beyond a configured retry budget, emits an `ingestion_incident` row visible to admins via the API.

### Doesn't

- Serve any HTTP traffic. The worker is reachable only by its scheduler and its database.
- Discover documents on its own. The curated set is the source of truth; adding documents is a config change owned by us, not an automatic crawl.
- Make tenant-aware decisions. Ingestion is corpus-global; it knows nothing about users, watchlists, or roles.
- Send alerts to clients. Pull-based dashboard surfacing only (notifications are deferred — see "Out of scope for the demo").

### How (high-level)

- Stateless worker process, one container, horizontally scalable if needed.
- Driven by a per-document schedule table (`document_id`, `next_poll_at`, `cadence`); each tick claims the due rows, polls them, updates `next_poll_at`. A simple `SELECT ... FOR UPDATE SKIP LOCKED` pattern keeps multiple worker replicas safe.
- Each per-document poll is a self-contained Postgres transaction. Either everything for that document (hash check, version row, blob upload, alignment, change events) commits, or none of it does.
- Alignment compute (shingling, MinHash, DP) lives in-process with the worker; no separate service.

## Public API service

The single HTTP surface every client talks to.

### Does

- Authenticates every request via bearer token (see "Authentication and user provisioning"); resolves the token to a `user_id`, a role (`client` or `admin`), and (for clients) the client's current subscription scope.
- Exposes the three primitives from doc 1 (discovery, temporal, differential) at corpus / document / clause scope, against the precomputed `change_events` table. **Every corpus query under the `client` role is filtered through the client's subscription scope before any other filter is applied.** A UK-only client receives zero rows for EU documents even on a corpus-wide query.
- Exposes **per-user state** endpoints: watchlists (CRUD), saved queries, dashboard payloads, alert preferences (preference storage exists even though delivery is deferred). Watchlist writes are validated against the client's subscription scope — a client cannot watch a document they aren't subscribed to.
- Exposes **subscription endpoints**:
  - For `client` role: read-only view of *their own* subscription.
  - For `admin` role: full CRUD over any client's subscription.
- Exposes **health and coverage** endpoints:
  - For `client` role: scoped to *that client's* subscription — last-scanned, last-change, change counts, coverage by jurisdiction/sector across what they're entitled to query.
  - For `admin` role: the same shape, plus system-wide views (ingestion health across all curated documents, scheduler status, recent `ingestion_incident` rows, aggregate stats), plus the ability to view any client's scoped view for support.

### Doesn't

- Ingest. The API is read-only against the corpus tables (`documents`, `document_versions`, `clauses`, `change_events`); it never calls Lawstronaut.
- Compute diffs at query time. Differential responses are assembled from precomputed `change_events` rows and the inline clause text on each side.
- Provide cross-client access at the `client` role under any circumstance. Admin-as-support views are a distinct, audited code path — not a flag toggled by the same endpoints.

### How (high-level)

- Thin HTTP layer over Postgres, with a clear seam between primitive endpoints (read-only, hot) and per-user endpoints (read/write, smaller payloads).
- **Tenant and subscription scoping use defence in depth.** Because both axes of isolation (cross-client privacy and subscription gating) are core architectural principles, neither relies on developer discipline alone.
  - **Private-state tables** (watchlists, saved queries, alerts, dashboards, subscriptions) are protected by Postgres **Row-Level Security (RLS)**. Each table carries a policy of the form `USING (user_id = current_setting('app.user_id')::uuid)`. The API resolves the bearer token to a `user_id` and sets `app.user_id` on the connection before any query runs. Even a raw SQL statement with a forgotten predicate is auto-scoped by Postgres.
  - A **repository / query-helper layer** wraps all private-table access in functions that take `user_id` as a required argument. Raw SQL against private tables in application code is lint-banned. This is belt-and-braces with RLS — RLS is the database-side guarantee, the repository pattern is the application-side guarantee.
  - **Corpus tables** (documents, change_events, etc.) under the `client` role are filtered by the client's subscription scope. The repository layer joins every corpus read against the requesting client's subscription (jurisdictions × sectors). An RLS policy on the corpus tables provides the second layer, using a security-definer scope function — even a corpus query that forgets the subscription join cannot return out-of-scope rows. The `admin` role connects with a session setting that bypasses subscription policies (its own audited code path).
  - **Multi-user integration tests** simulate two clients with disjoint subscriptions and assert (a) zero leakage between their private state and (b) zero leakage between corpus rows visible to each. These tests catch what policies and helpers cannot — payloads that correctly query *your own* data but inadvertently include another user's identifier, denormalised aggregates that leak counts across tenants, and similar logic-layer bugs.
- Pagination on corpus-scope responses, with the 3-second p95 budget from doc 3 as the operative constraint.
- Admin-role endpoints live under a distinct URL prefix (e.g. `/v1/admin/...`) so that the authorisation check at the edge is unambiguous and easy to audit.

## Webapp

A single-page application — our own UI, but architecturally a client.

### Does

- Authenticates against the API (login form → token exchange) and stores the token client-side for the session.
- Renders dashboards built from the public API:
  - **For clients:** everything is scoped to the client's subscription. Watchlist management (restricted to subscribed documents), coverage stats for subscribed and watched documents (last scanned, last change, # changes), change browsing with before/after clause diffs (driven by the differential primitive), saved-query management, read-only view of their own subscription.
  - **For admins:** everything clients see (without subscription gating), plus system-health views, subscription management for any client, and the support views (any client's dashboard, named explicitly so the UI never silently impersonates).
- Surfaces the alignment-confidence affordances described in doc 2 §"Change types and confidence" — defaulting to suppressing `MOVED` and below-threshold matches, with a toggle to include them.

### Doesn't

- Call the database directly. Ever.
- Have an "internal" API surface or any privileged channel to the backend services.
- Hold business logic beyond presentation. Auth, scoping, primitive computation, admin/client gating — all of it lives behind the API.

### How (high-level)

- Static bundle served from Azure Blob Storage (with CDN if needed); no SSR.
- HTTP/JSON only; same endpoints external customers consume.

## Authentication and user provisioning

- **Bearer tokens.** Every API request carries `Authorization: Bearer <token>`. Tokens resolve server-side to a `(user_id, role)` tuple, with the client's current subscription scope loaded alongside for client-role tokens.
- **Admin-provisioned accounts.** Users are created by us (the operator), not via self-signup. This matches the B2B sales motion and removes a large surface area (email verification, captcha, rate-limited signup) that the demo doesn't need.
- **Single login endpoint.** Both the webapp and programmatic customers obtain a token the same way — POST credentials, receive a token (and optionally a refresh token). One code path, one set of tests.
- **Token-to-user mapping** is the input to every tenant-scoping query in the API. If the token is missing, invalid, or expired, the request is rejected before any data is read.

## Out of scope for the demo

Named here so they don't get bolted on later as afterthoughts:

- **Notification service.** Email / webhook delivery on change events. Architecturally a fourth service that consumes `change_events`, joins to per-user alert preferences, and dispatches. Preference storage already exists on the per-user state endpoints; the consumer doesn't yet.
- **Self-service signup.** No public registration form. All accounts are operator-created.
- **Federated / SSO auth.** No OIDC, no SAML for now. Bearer tokens only.
- **Automatic upstream discovery.** Ingestion never reaches outside the curated set. New documents enter the system because an admin added them, not because Lawstronaut surfaced them.
- **Cross-document analytics** (e.g. "find every clause across the corpus matching a near-duplicate of this one"). The MinHash signatures persisted by ingestion (doc 2) make this cheap later, but it isn't a demo-day capability.

## Open questions

- **Subscription model granularity.** Starting position is "set of jurisdictions × set of sectors" — the cross-product of the two taxonomies, expressed as a stored filter on each subscription row. Open questions: do we need per-document overrides (a client gets a specific Act outside their normal jurisdictions, or excludes one inside them)? Per-document-version exclusions? Time-window limits ("subscription valid 2026-01-01 → 2026-12-31")? Default to the simple cross-product; add complexity only when a real contract demands it.
- **Admin-as-client support views: impersonation vs role bypass.** Should support views mint a short-lived scoped token for the target client (cleaner audit trail, harder to misuse), or be a role check on admin-prefixed endpoints (simpler)? Either works; pick before building the support flow.
- **Per-poll metrics aggregation.** Health-endpoint stats can be served by aggregating the raw per-poll telemetry rows at query time, or by maintaining a rollup (per-document daily counters) on ingestion commit. Decide once we know the polling volume.
- **Read replica trigger.** At what observed read latency or query mix do we add a Postgres read replica for the API? Set a watch threshold; don't add the replica preemptively.
- **Audit log scope.** Admin-as-support access to any client's data should leave an audit trail. Where does that log live — same Postgres, separate table, or a write-only sink? Tied to the impersonation-vs-bypass decision above.
- **Subscription change semantics.** When a client's subscription is reduced (e.g. they drop EU jurisdictions), what happens to their watchlist entries that are now out-of-scope? Auto-prune? Mark inactive? Keep but hide? Affects UX continuity vs strict-scope guarantees.
