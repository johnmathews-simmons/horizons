# Clause-tree parser

*Last revised: 2026-06-05.*

How raw markdown from a Lawstronaut content feed is turned into the immutable `Clause` tree that the alignment pipeline (doc 2) consumes. The parser is the bottom of the ingestion stack ‚Äî everything above it (similarity, alignment, change-event emission) assumes a `Clause` tree shaped the way this doc describes.

It is a prerequisite to `2. clause-alignment.md`. The alignment doc defines *what identity means across versions*; this doc defines *what one version looks like when parsed*.

## Scope

Pure markdown ‚Üí tree transform. No I/O, no DB, no `clause_uid` assignment, no version-to-version comparison. A consumer hands in markdown text and a `ParserConfig`, and gets a `Clause` root back. Two recogniser strategies cover the substrates the sample fixtures exhibit:

- **Heading-anchored** ‚Äî paragraph-leading markers (`PART N`, `**N.**`, `(N)`, `(a)`, `(i)`). Each paragraph contributes at most a handful of markers; structural depth is determined by the marker type.
- **Inline-numbered** ‚Äî markers embedded mid-paragraph in flowing prose (`ƒå√ÅST PRVN√ç`, `ƒål. I`, `N.` in Czech-style documents). Markers are recognised by the same pattern list, but a boundary check distinguishes structural anchors from citations.

The same pattern list and the same tree-building algorithm serve both. The substrate distinction is observational, not architectural.

## `Clause`

Immutable, hashable, recursive:

```python
@dataclass(frozen=True, slots=True)
class Clause:
    path: tuple[str, ...]            # human-readable address ("PART 1", "1", "(a)", "(i)")
    heading_text: str | None         # title text (from a preceding bold paragraph or a markdown heading)
    body_text: str                   # prose content of this clause only, not its descendants
    numbering_label: str | None      # the structural marker as it appeared ("PART 1", "1.", "(a)")
    children: tuple["Clause", ...]   # nested clauses, in document order
```

The root clause has `path=()`, `heading_text=None`, `body_text=""`, `numbering_label=None`. Its children are the document's top-level structural nodes. Helper `walk(root)` yields a DFS pre-order traversal.

`tuple` rather than `list` so the structure is hashable and safe to share between downstream pipeline stages. Downstream consumers that prefer a flat enumeration use `walk()` and ignore `children`.

## Recognisers

A `StructuralPattern` is `(name, regex, depth, requires_boundary)`. Patterns are tried in config order at each scan position; first-match wins. `requires_boundary=True` rejects matches that aren't at paragraph start or after sentence-terminating punctuation ‚Äî this is what keeps `(1)` recognised as a section anchor at the start of a paragraph but ignored when it appears inside a citation like "subsection (1)".

Default patterns (see `core/alignment/config.py`):

| Name | Regex | Depth | Boundary |
|---|---|---:|---|
| `ie_part` | `PART\s+(\d+)` | 1 | no |
| `cz_cast` | `ƒå√ÅST\s+\S+(?:\s+\S+)?` | 1 | no |
| `cz_clanek` | `ƒål\.\s+[IVXLCDM]+` | 2 | no |
| `section` | `(\d+[A-Z]?)\.` | 4 | yes |
| `subsection` | `\(\d+\)` | 5 | yes |
| `roman_subpara` | `\((?:i+\|[ivxlcdm]{2,})\)` | 7 | yes |
| `letter_para` | `\([a-z]\)` | 6 | yes |

Roman is tried before letter so `(i)` resolves as Roman depth-7 rather than Latin-letter depth-6. Single ambiguous chars (`(c)`, `(d)`, `(v)`, `(x)`) fall through to letter, accepting that a standalone Roman `(v)` after `(iv)` will mis-classify; the heuristic is right in the overwhelming majority of legal-doc cases.

Keyword patterns (`PART`, `ƒå√ÅST`, `ƒål.`) carry `requires_boundary=False` because they're unambiguous structural keywords ‚Äî accept them anywhere they appear. Paren and bare-number patterns require start-of-paragraph or a preceding `[.!?;:)"‚Äú‚Äù‚Äî]` followed by whitespace, which is the difference between a section anchor and an inline citation.

Depth gaps are allowed: a CZ document jumps from depth 2 (`ƒål. I`) straight to depth 4 (`N.`). The stack pops based on `depth >=`, so siblings at the same depth replace each other and deeper markers nest under the most-recently-opened ancestor.

## Pending headings

A bold-only paragraph that matches no structural pattern is treated as a *title* for the next-opened clause. If the currently-open clause has no `heading_text`, the bold text attaches there; otherwise it's parked as `pending_heading` and attaches to the next numbered clause. This is how Irish-style documents ‚Äî where a section title is a bold paragraph immediately preceding the bold number ‚Äî get titles into the tree.

The behaviour is gated by `ParserConfig.treat_unmatched_bold_as_heading`. Portals that don't follow this convention can disable it; the bold paragraph then becomes its own unlabelled leaf clause.

## Unmatched paragraphs

A paragraph with no structural markers and no bold-heading interpretation becomes its own leaf clause with `numbering_label=None`, child of the currently-open parent. This preserves atomicity for alignment: a stray paragraph between sections doesn't silently fold into the previous section's body, so a downstream content change to that paragraph is a discrete change event.

Prefix text before the first marker inside a paragraph receives the same treatment: a leaf with `numbering_label=None`.

## Configuration seam

`ParserConfig` is a Pydantic v2 model. The default is enough for the IE and CZ fixtures; per-portal override files arrive in WU2.1 and load YAML into the same shape. Nothing about the parser hard-codes a pattern list ‚Äî adding a new jurisdiction is a config change.
