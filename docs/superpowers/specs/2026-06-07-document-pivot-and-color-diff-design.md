# Document-pivot + colour-coded diff view

**Status:** draft (2026-06-07)
**Scope:** webapp + API. Two related UX changes for the demo. Numbered headings per project house style.

## 1. Goal

Two changes to the webapp:

1. From the homepage dashboard, clicking a jurisdiction or sector card should land on a paginated **table of documents** in that scope (not the recent-changes list).
2. The document split view should render the side-by-side panes with **every clause-level change visible and colour-coded by type** (added / removed / modified / moved), rather than highlighting only the single clause the user clicked through from.

## 2. Why

The headline demo moment is "what changed in this document and how." Today the card click goes to a flat list of change events, and the split view only highlights one clause at a time — both pull the viewer away from the document, not towards it. Pivoting the card click into a per-scope document table makes the document the unit of navigation; colouring every change in the split view makes the answer to "what changed?" visible at a glance instead of one row at a time.

## 3. Out of scope

- Sortable table headers.
- Column filtering / faceting beyond the existing search + jurisdiction + sector filters.
- Per-clause word-level diff inside MODIFIED clauses (the box + label is enough; word-level diff is a separate spec).
- Diffing arbitrary version pairs. The split view always compares the oldest version against the latest version, as today.
- New MCP / API surfaces beyond the additive `/v1/documents` fields.

## 4. Routing / information architecture

`HomeView.vue` currently branches:

```ts
function goToJurisdiction(code, changeCount) {
  if (changeCount > 0) router.push({ name: 'changes', query: { jurisdiction: code } })
  else router.push({ name: 'documents', query: { jurisdiction: code } })
}
```

Both `goToJurisdiction` and `goToSector` become unconditional pushes to `documents` with the appropriate query parameter. The `changeCount` argument can be dropped from the card `@select` event handlers. The "Browse recent changes" nav link is unaffected — users who want the flat changes list still have a single click to it.

## 5. Backend: `/v1/documents` extension

Add five fields to each item returned by `GET /v1/documents` and `GET /v1/documents/{id}`:

| Field | Type | Meaning |
|---|---|---|
| `clause_count` | `int` | Clause count of the latest version. `0` if the document has no versions. |
| `change_counts` | `{added, removed, modified, moved: int}` | Counts of change events whose `document_version_id` equals the latest version's id, grouped by `change_type`. All zero if the document has 0 or 1 versions. |
| `previous_version_at` | `datetime \| null` | `effective_date` (fallback `created_at`) of the second-latest version. `null` if fewer than 2 versions exist. |
| `current_version_at` | `datetime \| null` | Same for the latest version. `null` if no versions. |

### 5.1 Query shape

One additional aggregate per page (≤50 rows). Subquery selects the latest two `document_versions` per document by `effective_date desc nulls last, created_at desc`; left-joins `change_events` filtered by `document_version_id = latest_version_id` and groups by `change_type`. The existing RLS scope predicate is unchanged, so subscription gating still applies — no risk of cross-tenant leakage on the new fields.

### 5.2 Migration

None. Pure additive read query and response-model fields. The Pydantic `DocumentItem` schema gains the four fields with safe defaults.

### 5.3 OpenAPI / docs

The OpenAPI schema regen (`scripts/regen_endpoints_md.py`) will pick up the new fields automatically; `docs/api/endpoints.md` is regenerated by pre-commit.

## 6. Documents table view

Replace the `<ul>` in `DocumentsListView.vue` with a `<table>`. Keep the existing filter bar (search / jurisdiction / sector) and URL-sync behaviour.

### 6.1 Columns

| Header | Source | Rendering |
|---|---|---|
| Name | `title` | `RouterLink` to `/documents/{id}` |
| Length | `clause_count` | integer, right-aligned, plain `0` |
| Added | `change_counts.added` | integer, right-aligned, `0` rendered as muted `—` |
| Removed | `change_counts.removed` | same treatment |
| Modified | `change_counts.modified` | same treatment |
| Moved | `change_counts.moved` | same treatment |
| Previous version | `previous_version_at` | `YYYY-MM-DD`, empty if null |
| Current version | `current_version_at` | `YYYY-MM-DD`, empty if null |

### 6.2 Pagination

Page size **25** (smaller than today's 50 since the rows are wider). The query string carries both `limit` and `offset` so the page is shareable and reload-safe. Footer below the table: `Showing N–M of T` on the left, `‹ Prev · Page X of Y · Next ›` on the right. Prev / Next disabled at the ends. No numbered page list — unnecessary at demo scale.

### 6.3 State

Loading / error / empty states preserved from the current view, rendered as table caption / single-row variants so the layout doesn't jump.

## 7. Diff view with all changes colour-coded

In `DocumentDetailView.vue`, add a second query alongside the existing document query:

```ts
useQuery({
  queryKey: ['document-changes', documentId],
  queryFn: () => fetchDiscovery({ scope: 'document', document_id, limit: 200 }),
  enabled: computed(() => sortedVersions.value.length >= 2),
})
```

200 is comfortably above any demo document's change-event count; pagination is not in scope.

### 7.1 Per-pane change maps

Derive two lookup maps from the result and pass them down:

```ts
beforePathToType: Record<string, ChangeType>  // REMOVED + MODIFIED + MOVED → before_path
afterPathToType:  Record<string, ChangeType>  // ADDED   + MODIFIED + MOVED → after_path
```

`VersionPane` gains a new prop `changeMap: Record<string, ChangeType> | null` that **replaces** the current `highlightPath`. The left pane receives `beforePathToType`, the right pane receives `afterPathToType`. `ClauseOverlay` looks up each clause's `clause_path` in the map; if present, the clause renders inside a coloured box (the same border + background pattern as today's amber highlight) with a small pill in the top-right corner labelling the change type.

### 7.2 Colour palette

| Type | Border + background (Tailwind) | Pill |
|---|---|---|
| ADDED | `border-green-400 bg-green-50` | `bg-green-100 text-green-800` |
| REMOVED | `border-red-400 bg-red-50` | `bg-red-100 text-red-800` |
| MODIFIED | `border-amber-400 bg-amber-50` | `bg-amber-100 text-amber-800` (today's style) |
| MOVED | `border-blue-400 bg-blue-50` | `bg-blue-100 text-blue-800` |

A central constant (e.g. `src/constants/change-colors.ts`) maps `ChangeType → {boxClass, pillClass, label}` so the legend, `ChangeTypePill` (currently in `components/ui/change-type-pill/`), and `ClauseOverlay` all share the same source of truth.

### 7.3 Legend

A small horizontal strip between the document title block and the panes: four chips (coloured box + label). Rendered only when the changes query has any results, so single-version documents stay clean.

### 7.4 Deep-link compatibility

The existing `?before=…&after=…` query string from the Recent Changes list still works — but it now drives **only scroll-into-view** (the clicked clause anchors at the top of the viewport on arrival). All changes get coloured regardless of how the user arrived. The `highlightPath` prop on `VersionPane` is removed; if a `?before` / `?after` is present, `ClauseOverlay` scrolls that path into view on mount but the colouring comes from the change map.

### 7.5 Edge cases

- **0 versions** — no panes, the current "no content yet" state. No changes query fires (gated by `enabled`).
- **1 version** — single pane, no legend. Changes query stays disabled.
- **2+ versions** — two panes, legend, colours.
- **MOVED** — same blue box on both sides (the clause exists in both panes at different paths). On by default per user decision; matches the "all changes visible" goal.
- **Below-threshold alignment confidence** — the diff view does **not** filter by `alignment_confidence` (unlike Recent Changes, which has the "Show below-threshold" toggle). The intent here is "show every change the system thinks happened on this document"; confidence display is a follow-up if needed.

## 8. Component boundaries

The change touches one new constant and four existing files; nothing else needs to move.

| Unit | Responsibility | Dependencies |
|---|---|---|
| `src/constants/change-colors.ts` (new) | Maps `ChangeType → {boxClass, pillClass, label}`. Single source of truth for colours used by the diff view, legend, and existing change-type pills. | none |
| `HomeView.vue` (edit) | Unconditionally routes card clicks to the documents table. | router |
| `DocumentsListView.vue` (rewrite of the list body, keep filter bar) | Renders the paginated table and drives `limit` / `offset` from the URL. | `listDocuments` API, vue-query |
| `DocumentDetailView.vue` (edit) | Owns the second `useQuery` for change events; derives the two per-pane maps; renders the legend. | `fetchDiscovery`, vue-query |
| `VersionPane.vue` (edit) | Forwards `changeMap` to `ClauseOverlay`; the per-version clauses query is unchanged. | `getClauses`, `ClauseOverlay` |
| `ClauseOverlay.vue` (edit) | For each clause, looks up its path in `changeMap` and applies the matching colour box + pill. Handles scroll-into-view of the deep-link path on mount. | change-colors constant |

`ChangeTypePill` (already in `components/ui/change-type-pill/`) is refactored to read its colour classes from the new constant so the existing Recent Changes pills stay in sync with the diff view legend.

## 10. Risks and mitigations

- **Query cost on `/v1/documents` extension.** Adding two joins (latest-two-versions + change-event aggregate) to the list endpoint could regress p95 latency. Mitigation: the joins are bounded by page size (≤50 rows); the change-event aggregate uses the existing `(document_version_id, change_type)` index. If a perf check shows regression, fall back to two endpoints (option B in the brainstorm) — but only with evidence.
- **Visual noise on heavily-edited documents.** A document with many MOVED clauses (renumbering) could turn the right pane mostly blue. Decision was on-by-default per user. If demo feedback says it's too noisy, we add a toggle later — not now.
- **Deep-link semantics shift.** Today `?before=…&after=…` highlights the one clause; after this change it only scrolls into view. Anyone with an open bookmark sees more colour and the anchor clause centred. Behaviour is strictly additive — no regressions to surface.
